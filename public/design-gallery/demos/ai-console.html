Good catch. You are correct -- this is vulnerable to a timing attack. The === operator performs a byte-by-byte comparison and returns early on the first mismatch, which leaks information about how many leading bytes match. The fix: Use crypto.timingSafeEqual which performs a constant-time comparison:
typescript
import { timingSafeEqual } from 'crypto'; async function verifyToken(token: string, expected: string): Promise<boolean> { if (token.length !== expected.length) { return false; } const a = Buffer.from(token); const b = Buffer.from(expected); return timingSafeEqual(a, b); }
Additional recommendations:
1. The length check still leaks info. Consider hashing both values first with HMAC so length is always consistent.
2. Add rate limiting to your auth endpoint to further mitigate brute-force attacks.
3. Consider using a well-tested auth library like passport or jose instead of rolling your own token verification.
